Nantian Gateway 的控制面和数据面各自暴露一个 HTTP REST Admin API，用于运行时诊断、配置检查和健康探查。Admin API 是运维人员排查问题最常用的入口。

:::note
下方的 JSON 响应体仅为示意，展示大致结构；具体字段可能随版本变化。端点路径、方法和端口是权威信息。
:::

## 控制面 Admin API

控制面 Admin API 暴露在控制面 Pod 的 HTTP 端口上（默认 `18081`），提供集群级别的配置视图和数据面节点管理。

### 端点列表

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/livez` | 存活检查 |
| `GET` | `/readyz` | 就绪检查 |
| `GET` | `/v1/summary` | 集群配置概览（监听器、路由、后端和节点计数） |
| `GET` | `/v1/snapshot` | 当前推送给数据面的完整 IR 快照 |
| `GET` | `/v1/snapshot-sync` | 所有数据面节点的快照同步状态 |
| `GET` | `/v1/listeners` | 列出所有监听器 |
| `GET` | `/v1/routes` | 列出所有路由（HTTP/gRPC/TCP/UDP/TLS） |
| `GET` | `/v1/routes/{kind}/{namespace}/{name}` | 获取指定路由详情 |
| `GET` | `/v1/backends` | 列出所有后端集群 |
| `GET` | `/v1/backends/{namespace}/{name}` | 获取指定后端集群详情 |
| `GET` | `/v1/nodes` | 列出所有已连接的数据面节点 |
| `GET` | `/v1/nodes/{nodeId}` | 获取指定数据面节点的连接状态 |
| `POST` | `/v1/metrics/query` | 执行一次即时指标查询 |

### 健康检查

```bash
# 控制面存活检查
curl http://<controlplane-pod>:18081/livez
# 返回: 200 OK (body: "ok")

# 控制面就绪检查
curl http://<controlplane-pod>:18081/readyz
# 初始资源同步完成后返回 200 OK，否则返回 503 Service Unavailable
```

`/livez` 只检查进程是否存活，`/readyz` 额外检查控制面是否完成了初始资源同步，可以开始接受配置变更。

### 配置快照

```bash
# 获取各数据面节点的快照同步状态
curl http://<controlplane-pod>:18081/v1/snapshot-sync
# 返回示例（示意）:
# {
#   "version": "snap-20240601-120000-abc123",
#   "generated_at": "2024-06-01T12:00:00Z",
#   "nodes_synced": 3,
#   "nodes_total": 3
# }

# 获取完整配置快照
curl http://<controlplane-pod>:18081/v1/snapshot
# 返回当前推送给所有数据面的完整 IR 快照内容
```

配置快照端点返回的是控制面翻译后的内部表示（IR），不是原始的 Kubernetes 资源。这有助于排查配置翻译是否正确，以及确认数据面收到的配置内容。

### 数据面节点状态

```bash
# 列出所有已连接的数据面节点
curl http://<controlplane-pod>:18081/v1/nodes

# 返回示例（示意）:
# [
#   {
#     "node_id": "dataplane-7f8a9b-0",
#     "version": "v2026.06.0",
#     "connected_since": "2024-06-01T11:55:00Z",
#     "last_config_version": "snap-20240601-120000-abc123",
#     "last_config_nonce": "nonce-xyz789",
#     "last_config_status": "ACK"
#   }
# ]

# 获取指定节点详情
curl http://<controlplane-pod>:18081/v1/nodes/dataplane-7f8a9b-0
```

`/v1/nodes` 端点帮助确认数据面是否成功连接、收到了哪些配置版本、以及最后确认的状态。如果某个数据面长时间显示 `NACK` 状态，说明配置推送可能存在问题。

## 数据面 Admin API

数据面 Admin API 暴露在每个数据面 Pod 的 admin 端口上（默认 `19080`），提供单个代理实例的运行时状态。

### 端点列表

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/livez` | 存活检查 |
| `GET` | `/readyz` | 就绪检查 |
| `GET` | `/v1/summary` | 代理运行时概览 |
| `GET` | `/v1/node` | 本代理的节点标识和当前配置版本 |
| `GET` | `/v1/snapshot` | 当前应用的配置快照 |
| `GET` | `/v1/listeners` | 活跃监听器 |
| `GET` | `/v1/routes` | 活跃路由 |
| `GET` | `/v1/backends` | 后端集群与端点健康状态 |
| `GET` | `/v1/circuit-breakers` | 熔断器状态 |
| `GET` | `/v1/rate-limits` | 限流状态 |
| `GET` | `/v1/traffic` | 运行时流量统计 |
| `GET` | `/metrics` | Prometheus 指标端点 |

### 代理概览

```bash
# 获取代理运行时概览
curl http://<dataplane-pod>:19080/v1/summary

# 返回示例（示意）:
# {
#   "node_id": "dataplane-7f8a9b-0",
#   "version": "v2026.06.0",
#   "uptime_seconds": 86400,
#   "ready": true,
#   "config_version": "snap-20240601-120000-abc123",
#   "listeners": [
#     {"name": "http-80", "address": "0.0.0.0:80", "protocol": "HTTP", "active": true}
#   ]
# }
```

### 运行时统计

```bash
# 获取运行时流量统计
curl http://<dataplane-pod>:19080/v1/traffic

# 返回示例（示意）:
# {
#   "active_connections": 1250,
#   "requests_per_second": 45.2,
#   "total_requests": 3892000,
#   "connection_errors": 12
# }

# 获取后端集群与端点健康状态
curl http://<dataplane-pod>:19080/v1/backends

# 返回示例（示意）:
# [
#   {
#     "name": "backend-v1",
#     "namespace": "default",
#     "endpoints": [
#       {"address": "10.0.1.5:8080", "healthy": true},
#       {"address": "10.0.1.7:8080", "healthy": false}
#     ]
#   }
# ]
```

### 节点与配置版本

```bash
# 获取本代理的节点标识和已应用的配置版本
curl http://<dataplane-pod>:19080/v1/node

# 返回示例（示意）:
# {
#   "node_id": "dataplane-7f8a9b-0",
#   "config_version": "snap-20240601-120000-abc123",
#   "last_config_status": "ACK"
# }
```

## 常用诊断场景

### 场景一：确认配置是否已生效

当你在 Kubernetes 中修改了路由配置却看不到效果时，首先确认配置是否已推送并应用到数据面：

```bash
# 1. 检查控制面快照同步状态
curl http://<controlplane>:18081/v1/snapshot-sync

# 2. 检查数据面已应用的配置版本
curl http://<dataplane>:19080/v1/node

# 3. 如果版本不一致，检查数据面节点状态
curl http://<controlplane>:18081/v1/nodes
```

如果控制面快照版本已更新但数据面版本未更新，检查该节点是否处于 `NACK` 状态，以及错误详情中是否有配置解析失败的信息。

### 场景二：排查后端不可用

```bash
# 1. 检查控制面中后端集群的端点列表
curl http://<controlplane>:18081/v1/backends/default/backend-v1

# 2. 检查数据面中后端端点的健康状态
curl http://<dataplane>:19080/v1/backends
```

控制面返回的是从 Kubernetes EndpointSlice 翻译过来的配置，数据面返回的是实际健康检查的结果。如果控制面列出了端点但数据面显示 `healthy: false`，说明健康检查失败，需要检查后端服务本身是否正常。

### 场景三：检查数据面与控制面的连接

```bash
# 查看所有数据面的连接状态
curl http://<controlplane>:18081/v1/nodes | jq '.[] | {node_id, connected_since, last_config_status}'

# 查看特定数据面已应用的配置版本
curl http://<dataplane>:19080/v1/node
```

如果某个数据面始终无法同步到最新配置版本，说明控制面和数据面之间的 gRPC 连接可能不稳定，需要检查网络和防火墙配置。

:::tip[jq 工具]
Admin API 返回 JSON 格式，推荐使用 `jq` 工具格式化输出。如果执行环境没有 `jq`，可以用 `python3 -m json.tool` 代替。
:::
