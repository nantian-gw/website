The Nantian Gateway control plane and data plane each expose an HTTP REST Admin API for runtime diagnostics, configuration inspection, and health probes. The Admin API is the most commonly used entry point for operators troubleshooting issues.

:::note
The JSON response bodies below are illustrative examples of the shape you can expect; exact fields may vary by release. The endpoint paths, methods, and ports are authoritative.
:::

## Control Plane Admin API

The control plane Admin API is exposed on the control plane pod's HTTP port (default `18081`) and provides a cluster-level configuration view as well as data plane node management.

### Endpoint List

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/livez` | Liveness check |
| `GET` | `/readyz` | Readiness check |
| `GET` | `/v1/summary` | Cluster configuration overview (listener, route, backend, and node counts) |
| `GET` | `/v1/snapshot` | Full IR snapshot currently pushed to data planes |
| `GET` | `/v1/snapshot-sync` | Snapshot synchronization status across all data plane nodes |
| `GET` | `/v1/listeners` | List all listeners |
| `GET` | `/v1/routes` | List all routes (HTTP/gRPC/TCP/UDP/TLS) |
| `GET` | `/v1/routes/{kind}/{namespace}/{name}` | Get details for a specific route |
| `GET` | `/v1/backends` | List all backend clusters |
| `GET` | `/v1/backends/{namespace}/{name}` | Get details for a specific backend cluster |
| `GET` | `/v1/nodes` | List all connected data plane nodes |
| `GET` | `/v1/nodes/{nodeId}` | Get connection status for a specific data plane node |
| `POST` | `/v1/metrics/query` | Execute an instant metrics query |

### Health Checks

```bash
# Control plane liveness check
curl http://<controlplane-pod>:18081/livez
# Returns: 200 OK (body: "ok")

# Control plane readiness check
curl http://<controlplane-pod>:18081/readyz
# Returns: 200 OK when initial resource sync is complete, or 503 Service Unavailable
```

`/livez` only checks whether the process is alive. `/readyz` additionally checks whether the control plane has completed initial resource synchronization and is ready to accept configuration changes.

### Configuration Snapshot

```bash
# Get the snapshot synchronization status across data plane nodes
curl http://<controlplane-pod>:18081/v1/snapshot-sync
# Example response (illustrative):
# {
#   "version": "snap-20240601-120000-abc123",
#   "generated_at": "2024-06-01T12:00:00Z",
#   "nodes_synced": 3,
#   "nodes_total": 3
# }

# Get the full configuration snapshot
curl http://<controlplane-pod>:18081/v1/snapshot
# Returns the full IR snapshot content currently being pushed to all data planes
```

The configuration snapshot endpoints return the control plane's translated internal representation (IR), not the original Kubernetes resources. This helps troubleshoot whether configuration translation is correct and confirm what configuration the data plane has received.

### Data Plane Node Status

```bash
# List all connected data plane nodes
curl http://<controlplane-pod>:18081/v1/nodes

# Example response (illustrative):
# [
#   {
#     "node_id": "dataplane-7f8a9b-0",
#     "version": "v2026.06.0",
#     "connected_since": "2024-06-01T11:55:00Z",
#     "last_config_version": "snap-20240601-120000-abc123",
#     "last_config_nonce": "nonce-xyz789",
#     "last_config_status": "ACK"
#   }
# ]

# Get details for a specific node
curl http://<controlplane-pod>:18081/v1/nodes/dataplane-7f8a9b-0
```

The `/v1/nodes` endpoint helps confirm whether data planes have connected successfully, which configuration versions they have received, and their last acknowledged status. If a data plane shows a `NACK` status for an extended period, it indicates there may be a problem with configuration delivery.

## Data Plane Admin API

The data plane Admin API is exposed on each data plane pod's admin port (default `19080`) and provides the runtime state of a single proxy instance.

### Endpoint List

| Method | Path | Description |
|--------|------|-------------|
| `GET` | `/livez` | Liveness check |
| `GET` | `/readyz` | Readiness check |
| `GET` | `/v1/summary` | Proxy runtime summary |
| `GET` | `/v1/node` | This proxy's node identity and current configuration version |
| `GET` | `/v1/snapshot` | Currently applied configuration snapshot |
| `GET` | `/v1/listeners` | Active listeners |
| `GET` | `/v1/routes` | Active routes |
| `GET` | `/v1/backends` | Backend clusters and endpoint health |
| `GET` | `/v1/circuit-breakers` | Circuit breaker states |
| `GET` | `/v1/rate-limits` | Rate limit states |
| `GET` | `/v1/traffic` | Runtime traffic statistics |
| `GET` | `/metrics` | Prometheus metrics endpoint |

### Proxy Summary

```bash
# Get proxy runtime summary
curl http://<dataplane-pod>:19080/v1/summary

# Example response (illustrative):
# {
#   "node_id": "dataplane-7f8a9b-0",
#   "version": "v2026.06.0",
#   "uptime_seconds": 86400,
#   "ready": true,
#   "config_version": "snap-20240601-120000-abc123",
#   "listeners": [
#     {"name": "http-80", "address": "0.0.0.0:80", "protocol": "HTTP", "active": true}
#   ]
# }
```

### Runtime Statistics

```bash
# Get runtime traffic statistics
curl http://<dataplane-pod>:19080/v1/traffic

# Example response (illustrative):
# {
#   "active_connections": 1250,
#   "requests_per_second": 45.2,
#   "total_requests": 3892000,
#   "connection_errors": 12
# }

# Get backend clusters and endpoint health
curl http://<dataplane-pod>:19080/v1/backends

# Example response (illustrative):
# [
#   {
#     "name": "backend-v1",
#     "namespace": "default",
#     "endpoints": [
#       {"address": "10.0.1.5:8080", "healthy": true},
#       {"address": "10.0.1.7:8080", "healthy": false}
#     ]
#   }
# ]
```

### Node and Configuration Version

```bash
# Get this proxy's node identity and applied configuration version
curl http://<dataplane-pod>:19080/v1/node

# Example response (illustrative):
# {
#   "node_id": "dataplane-7f8a9b-0",
#   "config_version": "snap-20240601-120000-abc123",
#   "last_config_status": "ACK"
# }
```

## Common Diagnostic Scenarios

### Scenario 1: Verify Configuration Has Taken Effect

When you modify a routing configuration in Kubernetes but don't see the expected behavior, first confirm whether the configuration has been pushed to and applied by the data plane:

```bash
# 1. Check the control plane snapshot sync status
curl http://<controlplane>:18081/v1/snapshot-sync

# 2. Check which configuration version the data plane has applied
curl http://<dataplane>:19080/v1/node

# 3. If versions don't match, check data plane node status
curl http://<controlplane>:18081/v1/nodes
```

If the control plane snapshot version has been updated but the data plane version has not, check whether the node is in `NACK` status and whether the error details contain configuration parsing failure information.

### Scenario 2: Diagnose Backend Unavailability

```bash
# 1. Check the endpoint list for the backend cluster in the control plane
curl http://<controlplane>:18081/v1/backends/default/backend-v1

# 2. Check the health status of backend endpoints in the data plane
curl http://<dataplane>:19080/v1/backends
```

The control plane returns the configuration translated from Kubernetes EndpointSlice. The data plane returns the actual health check results. If the control plane lists endpoints but the data plane shows `healthy: false`, it means health checks are failing. You need to check whether the backend service itself is healthy.

### Scenario 3: Check Data Plane to Control Plane Connectivity

```bash
# View connection status for all data planes
curl http://<controlplane>:18081/v1/nodes | jq '.[] | {node_id, connected_since, last_config_status}'

# View the applied configuration version for a specific data plane
curl http://<dataplane>:19080/v1/node
```

If a data plane repeatedly fails to reach the latest configuration version, the gRPC connection between the control plane and data plane may be unstable. Check your network and firewall configuration.

:::tip[jq Tool]
The Admin API returns JSON format. Using the `jq` tool is recommended for formatting output. If `jq` is not available in your environment, you can use `python3 -m json.tool` as an alternative.
:::
